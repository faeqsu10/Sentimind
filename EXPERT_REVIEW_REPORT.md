## 🏗️ 1. 기술 아키텍처 검토 (Tech Architect)

### 1.1 현재 아키텍처 평가

**강점** ✅
- 단순성: 464줄 Express + 1,392줄 HTML → 배포 복잡도 극저
- Gemini 통합: 재시도 로직, 코드블록 파싱, HTTP 상태 처리 견고함
- OntologyEngine: 순수 클래스 (부작용 없음), 테스트 용이, 선언적 데이터
- 프론트엔드 UX: 탭 네비게이션, 접근성(ARIA), 반응형 디자인 우수

**약점** ⚠️
| 심각도 | 항목 | 문제 | 영향 |
|-------|------|------|------|
| **Critical** | 데이터 레이어 | JSON 파일 DB (O(n) 작업, 동시성 미지원) | ~1,000 entries에서 실패 |
| **Critical** | 인증 | 없음 (모든 데이터 공유) | 다중 사용자 불가능 |
| **High** | 프론트엔드 확장성 | 1,392줄 단일 HTML (코드 분할 없음) | 유지보수 어려움 |
| **Medium** | OntologyEngine | `findRelatedEmotions()` 항상 빈 배열 반환 | 한영 매핑 오류 |
| **Medium** | ID 생성 | `Date.now().toString(36)` (UUID 아님) | Supabase 마이그레이션 호환성 문제 |
| **Medium** | 상태 관리 | 저장 후 전체 entries 재조회 | 네트워크 오버헤드 증가 |

### 1.2 Phase 5 Supabase 마이그레이션 - 개선된 설계

#### 핵심 개선사항

1. **데이터 접근 레이어 추상화**
   ```
   현재: server.js에서 직접 JSON 파일 읽/쓰기
   개선: lib/json-store.js ↔ lib/supabase-store.js 인터페이스 공통화
   ```
   **효과**: Supabase 마이그레이션 시 drop-in replacement (USE_SUPABASE=true 환경 변수로 전환)

2. **점진적 마이그레이션 전략** (원본 JSON 유지)
   ```javascript
   // Phase 5: Supabase와 JSON 파일 동시 운영
   if (process.env.USE_SUPABASE === 'true') {
     return await supabaseStore.getEntries();
   } else {
     return await jsonStore.getEntries();
   }
   ```
   **이점**: 마이그레이션 중 롤백 가능

3. **스키마 수정**
   - `user_stats`, `activity_logs` 테이블 삭제 (Phase 5 범위 축소)
   - SQL 쿼리로 통계 계산 (별도 테이블 불필요)
   - `TIMESTAMP` → `TIMESTAMPTZ` (시간대 명시)

#### 수정된 Phase 5 일정
```
Week 1:
  - Day 1-2: 데이터 접근 레이어 추상화
  - Day 3-4: Supabase 초기화 + 스키마 생성
  - Day 5: RLS 정책 설정

Week 2:
  - Day 1-2: 마이그레이션 스크립트 (entries.json → PostgreSQL)
  - Day 3-4: API 통합 (USE_SUPABASE 전환)
  - Day 5: 통합 테스트

총 10일 (기존 12.5일 → 2.5일 단축)
```

### 1.3 프론트엔드 현대화 경로

**권고**: Vite + Vanilla TypeScript (React 미사용)

| 선택지 | 학습곡선 | TTM | 유지보수 | 추천 |
|-------|---------|-----|---------|------|
| 단일 HTML 유지 | 낮음 | 빠름 | 어려움 | ❌ Phase 1-4 (이미 한계) |
| Vite + 바닐라 TS | 중간 | 빠름 | 좋음 | ✅ Phase 6-7 추천 |
| Next.js | 높음 | 느림 | 좋음 | ❌ 초기 사용자 100명 규모에 오버 |
| React + Vite | 중간 | 보통 | 좋음 | ✅ 팀이 React 숙련시 고려 |

**Next.js가 필요한 이유**:
- ❌ SSR (서버 사이드 렌더링): 감정 일기 앱은 클라이언트-전용 앱
- ❌ SEO: 일기는 비공개, SEO 효과 미미
- ✅ API Routes: Express로 충분
- 결론: **Phase 7 Skip → 출시 후 고객 요청 시만 고려**
