## 1. Executive Summary

### 1.1 종합 평가

| 평가 영역 | 현재 점수 | 목표 점수 | 비고 |
|---------|---------|---------|------|
| 제품 비전 명확성 | 7/10 | 9/10 | 슬로건-기능 정렬 개선 필요 |
| 사용자 니즈 충족 | 6/10 | 9/10 | 핵심 가치(습관화) 부재 |
| 우선순위 합리성 | 5/10 | 8/10 | 기술 중심, 사용자 가치 역전 |
| 비즈니스 모델 | 6/10 | 8/10 | 무료 한도 근거 부족 |
| 로드맵 현실성 | 6/10 | 8/10 | 출시 시점 과도하게 지연 |
| 위험 관리 | 5/10 | 8/10 | 초기 사용자 확보 전략 미흡 |

### 1.2 가장 중요한 3가지 발견

**발견 1: MVP 출시 시점이 과도하게 지연되고 있다.**
- 현재 계획: 11주 후 (Phase 10) 마케팅 시작
- 실제 필요: Phase 5+6 완료 시점(약 5주)에 베타 사용자 모집 시작 가능
- 핵심 문제: 사용자 피드백 없이 개발만 진행하면 잘못된 방향으로 기술 투자 위험

**발견 2: 감정 분석 이후 "다음 단계"가 없다.**
- 현재: 감정 분석 → 위로 메시지 (여기서 끝)
- 사용자 기대: 감정 패턴 인식 → 구체적 행동 변화 → 삶의 개선
- 핵심 격차: 리텐션을 위한 "매일 돌아올 이유"가 현재 약함

**발견 3: Phase 5(Supabase 마이그레이션)는 사용자가 보이지 않는 작업이다.**
- 비즈니스 가치: 다중 사용자 지원 (Phase 6와 묶어서 의미 있음)
- 현재 Phase 5만으로 출시 가능한 새 기능: 없음
- 권고: Phase 5와 Phase 6를 단일 스프린트로 병합하여 최소 4주로 압축

## 2. 제품 비전 및 목표 명확성

### 2.1 현재 PRD 명확성 평가

**잘 된 부분:**
- 기능 요구사항(FR)에 수용 기준(AC) 명시됨
- 비기능 요구사항(NFR)에 구체적 수치 포함 (< 500ms, 99.5% uptime)
- Out of Scope 항목 명시로 스코프 크리프 방지 의도 있음
- 데이터 모델(스키마, RLS)이 상세함

**개선이 필요한 부분:**

| 문제 | 현재 상태 | 권고 사항 |
|------|---------|---------|
| 비전-기능 정렬 | "감정을 이해하고 함께 성장" vs 기능 목록 불일치 | "성장"을 지원하는 기능 추가 필요 |
| 사용자 여정 (User Journey) | PRD에 없음 | 일기 작성 → 분석 → 인사이트 → 행동의 플로우 명시 |
| 성공 정의 모호성 | DAU 100명, 리텐션 50% | "무엇이 성공인가"의 비즈니스 스토리 부재 |
| 요구사항 ID 체계 | FR-001~006, NFR-001~012 있음 | 우선순위(MoSCoW) 미표기 |

### 2.2 타겟 사용자 세분화 적절성

현재 세분화:
- Primary: 20-40대 직장인
- Secondary: 18-25 대학생
- Tertiary: 40+ 일반 대중

**문제점**: 세 그룹의 페인포인트와 동기가 완전히 다름에도 단일 제품 경험 제공 중

**권고 페르소나 (3가지):**

**페르소나 A - "번아웃 직장인" (Primary)**
- 이름: 김민준, 32세, 마케팅팀 대리
- 페인포인트: 감정 표현 못 하고 쌓임, 스트레스 원인 파악 어려움
- 핵심 동기: "왜 내가 자꾸 지치는지 알고 싶다"
- 사용 패턴: 퇴근 후 10시-11시, 5분 이내
- 요구 기능: 빠른 입력, 패턴 추적, 주간 리포트

**페르소나 B - "감정 성장 대학생" (Secondary)**
- 이름: 이소현, 22세, 대학교 3학년
- 페인포인트: 감정을 말로 표현하기 어려움, 일기 습관 없음
- 핵심 동기: "내 감정을 더 잘 이해하고 싶다"
- 사용 패턴: 수시로 짧게, 감정 급변 시
- 요구 기능: 쉬운 UI, 격려 메시지, SNS 공유 가능성

**페르소나 C - "자기성찰 추구자" (Tertiary)**
- 이름: 박영희, 45세, 워킹맘
- 페인포인트: 일기 쓸 시간이 없음, 감정을 체계적으로 정리하고 싶음
- 핵심 동기: "한 해 동안 내가 어떻게 성장했는지 보고 싶다"
- 사용 패턴: 저녁 취침 전, 15분+
- 요구 기능: 상세한 통계, 연간 리포트, 내보내기

**현재 제품의 최적 타겟**: 페르소나 A (번아웃 직장인) - 가장 명확한 페인포인트, 지불 의향 높음

### 2.3 MVP vs Phase 5-10 스코프 균형

**진짜 MVP란 무엇인가:**

MVP = 타겟 사용자가 핵심 가치를 경험할 수 있는 최소 기능 집합

Sentimind의 핵심 가치: "내 감정을 이해하고 패턴을 파악한다"

**현재 MVP 정의 문제:**
- Phase 4 완료 시점 이미 MVP 기능 완성됨 (일기 작성 + 감정 분석 + 통계)
- 그러나 Phase 5-9(11주)가 지나야 출시 계획
- 이 기간 동안 사용자 피드백 없이 기술 부채 누적 위험

**재정의된 MVP (Launch-Ready 기준):**

| 기능 | 현재 상태 | MVP 필요 여부 |
|------|---------|------------|
| 일기 작성 + 감정 분석 | 완료 | Must |
| 온톨로지 분류 | 완료 | Must |
| 공감 메시지 | 완료 | Must |
| 통계 대시보드 | 완료 | Must |
| 검색/필터 | 완료 | Should |
| 다중 사용자 지원 (Auth) | 미완료 | Must (출시 필수) |
| PostgreSQL 저장 | 미완료 | Must (다중 사용자 위해) |
| Next.js 마이그레이션 | 미완료 | Could (출시 후 가능) |
| 모니터링/Sentry | 미완료 | Should (출시 전) |
| CI/CD | 미완료 | Should (출시 전) |

**결론**: Phase 5(DB 마이그레이션) + Phase 6(인증)만 완료하면 실제 사용자에게 서비스 가능. Next.js 마이그레이션(Phase 7)은 출시 이후 진행 가능.

## 3. 기능 우선순위 재검토

### 3.1 현재 우선순위의 문제

현재 로드맵은 **기술 완성도** 순서로 설계되어 있음:
```
인프라 (Phase 5) → 인증 (Phase 6) → 프레임워크 (Phase 7) → 모니터링 (Phase 8) → 배포 (Phase 9) → 마케팅 (Phase 10)
```

이 구조의 문제:
- 사용자 가치 제공이 Phase 10까지 지연
- 배포 완료 전 사용자 피드백 0건
- 기술적으로 완벽하지만 시장 검증 없음

### 3.2 사용자 가치 기준 우선순위 재정렬

**MoSCoW 적용 (출시 기준):**

**Must Have (출시 필수):**
- M-01: 이메일 회원가입/로그인 (Supabase Auth)
- M-02: 사용자별 데이터 격리 (RLS)
- M-03: PostgreSQL 마이그레이션
- M-04: Gemini 감정 분석 (기존 완료)
- M-05: 일기 CRUD (기존 완료)
- M-06: Vercel 배포 + 도메인
- M-07: HTTPS + 기본 보안

**Should Have (출시 후 2주 내):**
- S-01: 에러 모니터링 (Sentry)
- S-02: 기본 로깅 시스템
- S-03: GitHub Actions CI/CD
- S-04: 일기 수정 기능 (24시간 내)
- S-05: 월간 감정 리포트 (이메일)

**Could Have (3개월 내):**
- C-01: Next.js 마이그레이션
- C-02: OAuth 로그인 (Google)
- C-03: 다크 모드
- C-04: 데이터 내보내기 (CSV)
- C-05: 주간 감정 인사이트 푸시 알림

**Won't Have (이번 Phase):**
- W-01: 소셜 기능 (공유, 친구)
- W-02: 모바일 앱
- W-03: 음성 인식
- W-04: B2B 기능
- W-05: 커뮤니티

### 3.3 Phase 순서 재설계

**현재**: Phase 5 → 6 → 7 → 8 → 9 → 10 (11주 후 출시)

**권고**:

```
[Week 1-4] Sprint A: 출시 준비
  - Supabase 설정 + RLS (기존 Phase 5)
  - Supabase Auth 통합 (기존 Phase 6)
  - Vercel 배포 (기존 Phase 9 일부)
  → 베타 출시 (유저 초대 10-30명)

[Week 5-6] Sprint B: 안정화
  - 사용자 피드백 반영
  - Sentry 모니터링 (기존 Phase 8)
  - CI/CD 파이프라인 (기존 Phase 9)
  → 정식 출시 준비

[Week 7-8] Sprint C: 성장 기능
  - 감정 패턴 주간 리포트
  - 이메일 리마인더 (습관화)
  → 공개 출시 + 마케팅

[Week 9-12] Sprint D: 기술 개선 (출시 후)
  - Next.js 마이그레이션 (기존 Phase 7)
  - 고급 분석 기능
  - 성능 최적화
```

**이 구조의 장점:**
- 4주 만에 실제 사용자 피드백 확보
- 7주 후 정식 출시 (현재 계획 대비 4주 단축)
- 사용자 피드백 기반으로 Phase D 내용 결정 가능

## 4. 사용자 가치 및 니즈 분석

### 4.1 현재 기능과 핵심 니즈의 정렬

**현재 제공 가치 레이어:**

```
Layer 1 (완료): 감정 표현 공간 - "오늘 있었던 일을 기록한다"
Layer 2 (완료): 감정 이해    - "내 감정이 무엇인지 AI가 알려준다"
Layer 3 (완료): 공감         - "누군가(AI)가 내 감정에 공감해준다"
Layer 4 (부분): 패턴 인식    - "내가 어떤 상황에서 자주 힘든지 안다"
Layer 5 (없음): 실행 인사이트 - "알게 된 패턴을 바탕으로 어떻게 해야 하나"
Layer 6 (없음): 성장 측정    - "시간이 지나면서 내가 나아지고 있는가"
```

**가장 시급한 보완 영역: Layer 5와 6**

현재 조언(advice) 필드가 있지만 "구체적이고 실행 가능한 제안"이 아닌 일반적 위로에 가까움.

### 4.2 반복 방문 유도 전략 (리텐션 엔진)

감정 일기 앱의 이탈 이유 Top 3 (업계 공통):
1. "쓸 것이 없다" - 습관 형성 실패
2. "써도 달라지는 게 없다" - 가치 인식 부족
3. "귀찮다" - 진입 마찰

**현재 Sentimind의 리텐션 도구:**
- 일기 목록 조회: 소극적 (다시 볼 이유가 약함)
- 통계 대시보드: 중간 (숫자가 있지만 인사이트가 없음)
- 공감 메시지: 일회성 (다음 날 다시 올 이유 아님)

**권고 리텐션 기능 (우선순위 순):**

**R-01: 연속 기록 스트릭 (Streak)** - 구현 난이도: 낮음 / 임팩트: 높음
- "7일 연속 기록 중" 배지 표시
- 스트릭 깨지면 "오늘 아직 안 쓰셨어요" 알림
- 근거: Duolingo의 핵심 리텐션 메커니즘과 동일

**R-02: 주간 감정 리포트** - 구현 난이도: 중간 / 임팩트: 높음
- 매주 월요일 "지난 주 당신의 감정" 요약 이메일
- 주요 감정 Top 3, 가장 힘들었던 날, 가장 좋았던 날
- "같은 주 기준 비교" (성장 느낌 제공)

**R-03: 감정 상세 인사이트** - 구현 난이도: 중간 / 임팩트: 중간
- "직장 관련 스트레스가 이번 달 3회 기록됨. 비슷한 시기 작년에도..."
- 패턴 → 구체적 행동 제안 연결
- 예: "번아웃 징후가 보입니다. 이번 주말 취미 활동을 추가해보세요"

**R-04: 비교 타임라인** - 구현 난이도: 낮음 / 임팩트: 중간
- "3개월 전 오늘 이런 감정을 느꼈어요" 알림
- 과거와 현재 비교로 성장감 제공
- 구현: created_at 기반 날짜 매칭

### 4.3 감정 패턴 → 실행 가능한 인사이트 전환

현재 Gemini 프롬프트가 생성하는 것:
- 감정 분류 (좋음)
- 공감 메시지 (좋음)
- 짧은 조언 (개선 필요)

권고 인사이트 강화 방향:

```
현재: "힘드셨겠어요. 잠시 쉬어가세요."

개선:
- 즉시 행동 제안: "5분간 심호흡이나 스트레칭을 해보세요."
- 패턴 연결 (3회 이상 동일 패턴): "이번 달 세 번째 직장 스트레스 기록입니다.
  주로 수요일에 집중되네요. 수요일 점심을 좀 더 여유롭게 계획해보시면 어떨까요?"
- 성취 강조: "지난주보다 긍정 감정 비율이 15% 올랐어요."
```

## 5. 비즈니스 모델 검증

### 5.1 프리미엄 모델 적절성 평가

현재 계획된 모델:
- 무료: 월 30개 일기
- 프리미엄: $4.99/월 (무제한)
- Pro: $9.99/월 (무제한 + 고급 분석)

**월 30개 한도의 문제점:**

목표 사용자가 매일 일기를 쓴다면:
- 30일 × 1회 = 30개 (무료 한도 소진)
- 즉, 진정한 헤비 유저는 첫 달부터 프리미엄 필요

이 설계의 위험:
- "매일 쓰는 습관"을 유도하면서 한 달 후 유료 전환 강제
- 사용자 불만 가능성: "왜 일기를 많이 쓰면 돈을 내야 하나"
- 초기 단계에서 습관 형성을 방해할 수 있음

**대안 프리미엄 모델 (권고):**

| 플랜 | 현재 제안 | 권고 수정안 | 이유 |
|------|---------|-----------|------|
| 무료 | 월 30개 | 무제한 일기 + 기본 통계 | 습관 형성 우선 |
| 프리미엄 ($4.99) | 무제한 | 무제한 + 주간 리포트 + 인사이트 | 기능 차별화 |
| Pro ($9.99) | 고급 분석 | 프리미엄 + 연간 리포트 + 내보내기 + 우선 지원 | B2C 상위 플랜 |

**근거**: 일기 개수가 아닌 "분석 깊이"로 차별화하는 것이 사용자 가치에 더 부합.

### 5.2 Gemini API 비용 재검토

현재 예상 계산:
- DAU 100명 × 2회/일 = 200회/일
- 월간 6,000회 × $0.075/1K tokens = $0.45/월

**실제 비용 리스크:**
- Gemini 2.5 Flash 요금: 입력 $0.075/1M tokens (2026-03 기준)
- 일기 텍스트 평균 500자 ≈ 200-300 tokens
- 시스템 프롬프트 포함 ≈ 1,000-2,000 tokens/요청
- 월 6,000회 × 1,500 tokens = 9,000,000 tokens = $0.675/월

비용 자체는 낮으나 무료 플랜으로 운영 시:
- DAU 500명 × 2회 × 30일 = 30,000회/월
- 30,000 × 1,500 tokens = 45M tokens = $3.38/월

즉, **실제 비용은 매우 낮으므로 월 30개 한도는 비용 절감이 아닌 전환율 목적**.

**권고**: 첫 6개월은 무제한 무료로 운영하여 사용자 데이터 확보 후 모델 결정.

### 5.3 수익화 시점

**권고 수익화 타임라인:**

```

  목표: 첫 500명 사용자 확보
  측정: 사용자당 평균 월 일기 수 (실제 데이터 기반 한도 결정)
```

### 7.1 주요 경쟁 서비스 분석

### 7.2 Sentimind의 실제 차별화 포인트

**차별화 1: 온톨로지 기반 3단계 감정 분류**
- 경쟁사: 단순 긍정/부정/중립 or 이모지 선택
- Sentimind: 대분류(긍정) → 중분류(기쁨) → 소분류(설렘) 자동 분류
- 사용자 인식 가능성: 높음 (시각적으로 명확)

**차별화 2: 상황 맥락 자동 감지**
- 경쟁사: 태그 수동 입력
- Sentimind: 직장/학업/건강/관계/자기성찰 자동 분류
- 사용자 인식 가능성: 중간 (이모지로 표시되나 설명 부족)

**차별화 3: Gemini 기반 개인화 공감 메시지**
- 경쟁사: 템플릿 기반 응원 메시지
- Sentimind: 텍스트 맥락에 맞는 공감 생성
- 사용자 인식 가능성: 높음 (실제 차이 체감 가능)

### 7.3 사용자가 인지 가능한 마케팅 메시지 (권고)

현재 슬로건: "당신의 감정을 이해하고, 함께 성장하는 일기"
- 문제: "함께 성장"이 추상적, 차별점 미포함

**권고 메시지 (A/B 테스트 후 결정):**

옵션 A (분석 강조): "한 줄만 써도 AI가 당신의 감정을 3단계로 분석해요"
옵션 B (습관 강조): "매일 한 줄, 6개월 후 당신의 감정 지도"
옵션 C (공감 강조): "판단 없이, AI가 오늘 당신의 마음을 들어드려요"

**권고**: 출시 시 A 사용 (기능 차별화 명확), 리텐션 확보 후 B로 전환.

### 8.3 Phase 7 (Next.js 마이그레이션) 재검토

Next.js 마이그레이션의 비용-효과 분석:

**비용:**
- 2주 개발 시간 (Frontend Dev)
- 학습 곡선 (팀이 Express에서 Next.js로 전환)
- 마이그레이션 중 기능 추가 불가

**효과:**
- SSR (Server-Side Rendering) 지원 → SEO 개선
- API Routes 통합 → 코드 단순화
- Vercel 최적화 배포

**결론**: Phase 7은 출시 후로 연기해도 무방.
- Express + 단일 HTML 현재 스택으로 충분히 서비스 운영 가능
- 사용자 100명 이하에서 SSR 필요성 없음
- SEO는 감정 일기 특성상 콘텐츠 기반 트래픽이 없음 (개인 앱)

**권고**: Phase 7을 3개월 후로 연기하고, 해당 2주를 리텐션 기능에 투자.

### 11.1 이번 주 내 결정해야 할 사항

**Decision 1: 팀 규모 확정** (기한: 이번 주)

- 11주 계획에서 2주 절약 가능
- 대신 리텐션 기능(스트릭, 주간 리포트) 개발에 투자
